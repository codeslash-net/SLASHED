Buildless CSS framework · 201 tokens · 647 utility classes · 50+ components · README · Docs

Philosophy: UI components are .cs-* prefixed. Layout primitives and utilities are unprefixed for ergonomic authoring. Framework uses cascade layers — any unlayered BEM class you write beats framework rules automatically.

Three authoring layers: (1) Layout primitives (.section, .stack, .bento) encode structural patterns with responsive behaviour built-in. (2) BEM components (.cs-btn, .cs-card) encode design decisions with design tokens. (3) Utilities configure the layout engine, toggle a11y properties, and compose inline typography — they do not encode decoration.

Design Tokens tokens-default.css · see docs/TOKENS.md

System architecture: Tokens form a three-tier hierarchy. Primitives are the raw scale values (--primary-100, --space-m). Semantic aliases give role-based names to primitives (--color-bg, --color-text) — components and utilities always reference aliases, never raw primitives. Instance tokens are component-scoped overrides (--btn-bg, --card-padding) that let you customise one instance without touching the cascade.

How to configure: Override primitives on :root in your own stylesheet. The entire system (shades, dark mode, components) updates automatically. Do NOT reach into component CSS or utility rules — token override is the correct customisation path.

Brand Colors — primitives (6 configurable roots)

Each of the 6 brand roles is an oklch() value registered via @property (enables CSS transitions between themes). The full shade scale (50–950), hover tint, and 3 alpha variants are derived via color-mix(in oklch, …) — change one root and all 15+ derived values update. Dark-mode shade inversion is automatic. Use status colors (--success, --warning, --error) for feedback states, not --primary tinted to look greenish.

--secondarySecondary brand color. Default: #0d9488.Used for supporting actions, links, tags. Same derivation system as primary.
--accentAccent / highlight color. Default: #f59e0b.Warm contrast against cool primary. Used for highlights, callouts, badges.
--success / --warning / --errorStatus feedback colors.Semantic, not brand. Always use these for feedback states — never primary-tinted hues.
--primary-50 … --primary-950Primary shade scale (10 stops: 50, 100, 200, 300, 400, 600, 700, 800, 900, 950).Derived via color-mix(in oklch). Use for tinted backgrounds, hover fills, disabled states.
--primary-a25 / -a50 / -a75Alpha variants at 25%, 50%, 75% opacity.For overlays and tinted surfaces where transparency is needed. Same pattern for secondary and accent.
--primary-hoverHover state tint (slightly darker/lighter than root).Computed from root — guarantees hover always contrasts with base regardless of custom brand color.

Neutral scale

Neutrals are derived via color-mix(in oklch, var(--primary) var(--neutral-tint), gray). With --neutral-tint: 0% (default) you get pure gray. Set --neutral-tint: 2-3% for brand-cohesive grays that match your primary hue. ⚠ issue Default of 0% makes the tinting feature invisible — a consumer who doesn't read docs never discovers it. Decision needed: change default to 2% or keep 0% with a more visible note.
--neutral-tintControls brand-cohesion of gray scale. Default: 0% (pure gray).Set to 2–3% for grays that feel "on-brand". Low-impact but high-polish detail.
--neutral-50 … --neutral-90010-stop neutral scale, light to dark.Used for backgrounds, borders, disabled states, text hierarchy. Flip in dark mode automatically.

Semantic color aliases

These aliases decouple component authoring from specific shade values. A component referencing --color-bg works correctly across light/dark mode and across any brand color, because these aliases are redefined in the dark mode blocks. Never use raw shade tokens (--neutral-100, --primary-50) directly in component BEM rules — use semantic aliases so the component responds to theme changes.
--color-bgPage / root background. Light: near-white. Dark: near-black.The baseline. Every surface sits on top of this.
--color-surfaceCards, modals, elevated elements. One step above --color-bg.The most-used surface. Components default to this.
--color-surface-2Nested surfaces, hover backgrounds. One step above --color-surface.For hover states and nested cards — creates clear elevation hierarchy.
--color-surface-elevatedDropdowns, tooltips, floating panels.Highest surface elevation — creates clear separation from page content.
--color-surface-overlayModal backdrop overlay.Dedicated token so consumers can change backdrop opacity/color independently.
--color-textPrimary body text.The default. All base text renders through this.
--color-text-secondarySubheadings, secondary labels.One step below primary — for slightly de-emphasised but still readable text.
--color-text-mutedHints, placeholders, metadata.Visibly subordinate but still accessible. Passes WCAG AA for large text.
--color-text-faintDecorative labels, watermarks. ~3.4:1 contrast.⚠ contrastFails WCAG AA for body text. Decision: rename to --color-text-decorative to make the intent explicit, or add a prominent "decorative only" warning.
--color-text-disabledDisabled form controls. ~2.7:1 contrast (WCAG exemption for disabled).WCAG explicitly exempts disabled controls from contrast requirements. Value is intentional.
--color-text-on-primaryText on primary-colored backgrounds.Auto-computed via Relative Color Syntax (oklch(from …)) in supporting browsers. Static fallback for others.
--color-text-on-darkText on dark/inverted surfaces.For footer-dark, hero-dark, and any .bg-dark context.
--color-borderDefault border color.Used by --border-width components and .border visual utility.
--color-border-subtle / --color-border-strongWeaker and stronger border variants.For dividers (subtle) and input focus rings (strong). Provides contrast hierarchy without custom values.
--color-link / --color-link-visited / --color-link-hoverLink color stack.Centralises link styling — override once, affects all unclassed links.

Spacing scale — fluid clamp (11 steps + semantic aliases)

All spacing tokens are clamp(min, fluid, max) computed across 320–1440px viewport. You never set px values directly — always use tokens. Each step is approximately 1.5× the previous. The scale maps to typical component padding, gap, and layout spacing needs. Semantic aliases (--space-section, --space-content, --space-gap, --space-gutter) provide named slots for the most common layout roles. Spacing utilities (.p-m, .gap-l) are for layout configuration. For component-intrinsic spacing, use instance tokens (--card-padding, --btn-padding) that default to spacing tokens.
--space-px1px (fixed)Hairline — borders, separators. Not fluid because 1px is always 1px.
--space-0-5~2pxSub-pixel gap — icon padding, tight chip spacing.
--space-2xsclamp(0.25rem → 0.375rem)Micro spacing — badge padding, tight label gaps.
--space-xsclamp(0.5rem → 0.75rem)Small intra-component gaps — button icon gap, form hint margin.
--space-sclamp(0.75rem → 1.125rem)Default component internal padding (inputs, small cards).
--space-mclamp(1rem → 1.5rem)Standard gap / padding. The most-used step.
--space-lclamp(1.5rem → 2.25rem)Card padding, section sub-gaps.
--space-xlclamp(2rem → 3rem)Section padding, large card padding.
--space-2xlclamp(3rem → 4.5rem)Between major sections inside a page block.
--space-3xlclamp(4rem → 6rem)Page-level vertical rhythm.
--space-4xlclamp(6rem → 9rem)Hero/section top/bottom padding. Large breathing room.
--space-section= --space-3xlSemantic alias for .section padding. Override to globally change section rhythm.
--space-content= --space-xlInner content padding (inside .container).
--space-gap= --space-mDefault grid/flex gap. Override to globally tighten or loosen all grids.
--space-gutterfluid page gutterHorizontal page margin — used by .px-gutter. Override for custom edge spacing.

Typography scale — fluid clamp (10 body steps + hero)

Same fluid philosophy as spacing. All sizes use clamp(min, fluid, max) across 375–1600px. No breakpoint-triggered font size changes — the scale is continuous. --h1..h6 are aliases pointing to size-scale tokens: headings reuse body scale steps for coherence. --text-fluid-hero has a wider range and higher ceiling — reserved for single large marketing headings. Don't use --text-fluid-hero on body text or section subheadings — it's very large at wide viewports.

--text-lclamp(~20px → ~24px)Lead paragraphs, intro text, blockquote.
--text-xlclamp(~24px → ~30px)h5/h4 level headings, card titles.
--text-2xlclamp(~28px → ~36px)h3 level headings, feature section titles.
--text-3xlclamp(40px → 54px)h2 level headings, major section titles.
--text-4xlclamp(~52px → ~72px)h1 level, page titles.
--text-5xlclamp(~64px → ~96px)Large page titles, splash screens.

Layout Primitives slashed-core.css · unprefixed

Layout primitives are the primary building blocks. Unlike utility classes, they encode structural patterns with responsive behavior, semantic spacing, and container-awareness built in. Reach for a primitive first — use utilities only for one-off adjustments on top. Grids in SLASHED use @container queries (not viewport media queries) so they respond to their own width, not the page width. Do NOT use layout primitives as BEM components — they have no cs- prefix intentionally. Don't apply BEM modifiers to them either; use instance tokens or modifier classes documented below.

UI Components .cs-* prefixed · docs/COMPONENTS.md

Why BEM components: Components encode entire patterns — their layout, states, tokens, and accessibility semantics in one class. Unlike utilities, they are self-contained: you don't stack 8 utility classes to get a button. Components use instance tokens for single-line customisation without !important. Never write into @layer slashed.overrides — that layer is reserved for consumers. To customise a component: (1) set its instance token via style="", (2) add a BEM modifier --variant, or (3) write an unlayered .my-btn { … } rule — it automatically beats all framework layers.

Spacing Utilities

Why this family: All spacing utilities use the fluid --space-* token scale — values derived with clamp(min, fluid, max) between 320px and 1440px viewports. This means responsive spacing is built in automatically; you never write @media rules just to adjust padding or gap. Use spacing utilities for contextual overrides. If padding or gap is a fixed part of a component's design (e.g. .cs-card always has --card-padding), change the instance token instead: style="--card-padding: var(--space-2xl)". Writing .p-2xl on every card instance is noise.

Responsive Variants sm: 30em · md: 48em · lg: 64em · xl: 80em

Why breakpoint prefixes: Responsive variants handle discrete layout switches — changing direction, showing/hiding, switching column counts. You shouldn't need them for spacing or typography because those are already fluid via clamp() tokens. The breakpoint values are hardcoded in slashed-utilities.css (CSS custom properties can't be used inside @media). Don't use breakpoint utilities for spacing changes — fluid tokens handle that automatically. Don't use them inside layout primitives that use @container — those respond to their own width, not the viewport. Use breakpoint utilities only for viewport-level layout switches (show/hide a mobile menu, collapse columns to single).

Accessibility, Print, Dark Mode

Why this section exists: These utilities live in @layer slashed.a11y — the highest-specificity layer in the framework. They override everything below without !important. This is intentional: accessibility and user preferences must never lose to component or utility rules. Don't use .dark:* utilities as a substitute for proper semantic dark-mode tokens. If a component should adapt to dark mode, it should use light-dark() in its token definitions — not require a .dark:* override on every instance.

Accessibility

.sr-onlyVisually hidden but accessible to screen readers (clip + absolute)The standard accessible-label pattern. Use for icon-only buttons, skip links, and extra context for AT users. Does NOT remove from tab order.
.not-sr-onlyReverses .sr-only — restores visual displayFor focus-visible skip links that should appear on keyboard focus but stay hidden otherwise.
.skip-to-contentSkip-to-main-content link (visible on focus only)WCAG 2.4.1 (Level A) requirement — allows keyboard users to skip repeated navigation. Required on every page.
.has-invalidApplies error styles when a child input is in :user-invalid stateFor styling the wrapper container when an input is invalid — :has(:user-invalid) — without JS. Works with .cs-form-group.
.has-checkedApplies styles when a child checkbox/radio is :checkedFor custom toggle cards, selection states, and filter groups. Native :has(:checked) replaces JS toggle-class patterns.
.scroll-mt-header / .scroll-pt-headerOffset scroll targets below sticky header (uses --header-height)Prevents anchor links from landing under the sticky navbar. Required when using scroll-behavior: smooth and a sticky header.
